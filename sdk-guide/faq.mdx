## General

<AccordionGroup>
  <Accordion title="What is the package name?">
    **`vaults-multichain-sdk`** on npm. Main export: **`TrustodianSDK`**. Default import:

    ```typescript
    import { TrustodianSDK, EChainType, InMemoryShareStore } from 'vaults-multichain-sdk';
    ```
  </Accordion>

  <Accordion title="What chains does the SDK support?">
    **EVM** (Ethereum, Base, Arbitrum, Optimism, Polygon, BNB, Avalanche + testnets), **Solana**,
    **Tron**, **Bitcoin** (`BTC` / `BTC_T3`), **Cardano**, and **XRP**.
  </Accordion>

  <Accordion title="SDK vs raw REST API?">
    The SDK wraps `/sdk/*` endpoints and adds OPRF, encrypted share decryption, Shamir
    reconstruction, and chain-specific signing. Raw HTTP is possible but you must implement all
    crypto yourself. **Use the SDK.**
  </Accordion>

  <Accordion title="Can I run the SDK in the browser?">
    Possible with a `buffer` shim, but **not recommended**. Run on your backend; never expose API
    keys or wallet passwords to clients.
  </Accordion>
</AccordionGroup>

## Keys & Security

<AccordionGroup>
  <Accordion title="How does key management work in v2?">
    1. Integrator implements **`shareStore`** (device share S0)
    2. User sets a **wallet password** (`setWalletPassword` / `unlockWalletSession`)
    3. SDK fetches **encrypted** server/recovery shares from API
    4. SDK decrypts and reconstructs key in memory for signing
    5. Key is zeroized after use

    See [Key Management](/sdk-guide/key-management).
  </Accordion>

  <Accordion title="What is userRef?">
    The Trustodian user id for your API key — from `await sdk.getSdkUserId()`. It keys `shareStore`
    entries, wallet-password sessions, and crypto calls.
  </Accordion>

  <Accordion title="What is the wallet password?">
    A user-chosen secret that encrypts server/recovery shares. Set once with `setWalletPassword`,
    then pass on each session via `unlockWalletSession`. Required before `createWallet`, `sendTransaction`,
    `signOrder`, or any key operation. Never send it to client apps or log it.
  </Accordion>

  <Accordion title="What are client environments (web vs sdk)?">
    Trustodian stores encrypted shares separately per environment (`web`, `mobile`, `extension`, `sdk`).
  The Node SDK always uses **`sdk`**. Your `shareStore` holds the device share for that environment only.
  Dashboard wallets and SDK wallets for the same account are independent — each needs its own password
  setup and device share.
  </Accordion>

  <Accordion title="Can Trustodian access my users' funds?">
    **No.** Trustodian holds at most one encrypted share. Spending requires the wallet password +
    device share (under your control) or recovery path.
  </Accordion>

  <Accordion title="What is KeyMismatchError?">
    Device share version is stale. Pass `walletPassword` on the crypto call for automatic recovery,
    or call `recoverWallet` / `recoverWalletsBatch`.
  </Accordion>
</AccordionGroup>

## Wallets

<AccordionGroup>
  <Accordion title="Can I import an existing wallet?">
    ```typescript
    await sdk.importWallet({
      chain: EChainType.EVM,
      privateKey: '0x...',
      userRef,
      password,
    });
    ```
    The key is split locally — **not** sent to the server.
  </Accordion>

  <Accordion title="createWallet modes: generate vs ecdh?">
    - **`generate`** — local keygen (default when `genWalletFromMnemonic` is false)
    - **`ecdh`** — backend derives from Vault mnemonic, delivers via ECDH (when flag is true)

    Check `await sdk.getAppConfig()`.
  </Accordion>

  <Accordion title="Why do delete/rename need chain + address?">
    Wallets are keyed by `(chainType, address)` on the API. Always pass both to `deleteWallet` and
    `updateWalletName`.
  </Accordion>
</AccordionGroup>

## Organization

<AccordionGroup>
  <Accordion title="How do org wallets work?">
    API key resolves to an organization context. Use org list/create methods (`listOrgWallets`, etc.)
    and pass `walletScope: 'organization'` on send/sign/order methods. See
    [Organization](/sdk-guide/organization).
  </Accordion>

  <Accordion title="How do employees get wallet access?">
    Master calls `provisionOrgMemberWalletAccess`; employee calls
    `acceptOrgWalletAccessWithPassword`. Device share is stored in your `shareStore` after accept.
  </Accordion>
</AccordionGroup>

## Multisig & Orders

<AccordionGroup>
  <Accordion title="Do I set vault name on createMultisig?">
    **No.** Create returns the vault; call `updateMultisigName` afterward.
  </Accordion>

  <Accordion title="Are estimateActivationFee / estimateOrderExecutionFee accurate?">
    Currently return **placeholder** values (`0.001`). Use chain RPC or your own estimation for
    production UX.
  </Accordion>
</AccordionGroup>
