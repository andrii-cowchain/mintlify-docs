The Cowchain SDK (`vaults-multichain-sdk`, also referred to as `TrustodianSDK`) is a TypeScript library built for **server-side integrators**. It wraps the Trustodian `/sdk/*` REST API and handles everything from encrypted Shamir share management to chain-specific transaction signing — all without the private key ever existing in one place at the same time. Whether you are building a custody-free exchange, a B2B payment service, or a multi-chain portfolio product, the SDK lets you manage wallets across six chains through a single, consistent interface.

## What Is the Non-Custodial 2-of-3 Model?

Cowchain ensures that **no single party ever holds a complete private key**. When a wallet is created, the SDK splits the private key into three mathematically related shares using **Shamir's Secret Sharing**. Any two of those three shares are sufficient to reconstruct the key for signing — but one share alone is cryptographically useless.

| Share | Held by | How it is protected |
| --- | --- | --- |
| **Device Share (S0)** | You, via your `shareStore` implementation | Stored in your own database; your responsibility |
| **Server Share (S1)** | Trustodian API | AES-GCM encrypted with a key derived from the wallet password via OPRF |
| **Recovery Share (S2)** | Trustodian crypto service | AES-GCM encrypted with a second key + Vault Transit |

Signing happens **inside the SDK process, in memory**. The SDK fetches the encrypted server share, combines it with your device share, reconstructs the key, signs the transaction, and immediately zeroizes the key from memory. Trustodian's servers never see the reconstructed private key; your servers never store it at rest.

<Note>
  Read-only calls such as listing wallets or fetching portfolio data require only your API key. Any
  operation that **touches keys** — create, send, sign — additionally requires a `shareStore`
  implementation and an active wallet-password session. See [Key Management](/sdk-guide/key-management)
  and [Authentication](/sdk-guide/authentication).
</Note>

| Chain | `EChainType` | Networks |
| --- | --- | --- |
| **EVM** | `EVM` | Ethereum, Base, Arbitrum, Optimism, Polygon, BNB, Avalanche (+ testnets) |
| **Solana** | `SOL` | Mainnet / devnet |
| **Tron** | `TRON` | Mainnet / Shasta |
| **Bitcoin** | `BTC` / `BTC_T3` | Mainnet / testnet — use `getBtcChainType()` on testnet |

| **XRP** | `XRP` | Mainnet / testnet |

## Installation

Install the SDK with your preferred package manager:

<CodeGroup>

```bash npm
npm install vaults-multichain-sdk
```

```bash yarn
yarn add vaults-multichain-sdk
```

```bash pnpm
pnpm add vaults-multichain-sdk
```

</CodeGroup>

<Warning>
  **Node.js backend only.** Never ship the SDK to a browser or mobile client — your API key and
  wallet passwords must remain on your server. Node 18+ includes `Buffer` globally; no extra
  polyfills are needed.
</Warning>

## Quick Start

The snippet below shows the minimal flow: initialize the SDK, set a wallet password for the first time, and create an EVM wallet.

```typescript
import { TrustodianSDK, InMemoryShareStore, EChainType } from 'vaults-multichain-sdk';

// Initialize once per server process (or per request in stateless environments)
const sdk = new TrustodianSDK({
  apiKey: process.env.TRUSTODIAN_API_KEY!,
  network: 'testnet',           // 'mainnet' | 'testnet'
  shareStore: new InMemoryShareStore(), // replace with your DB adapter in production
});

// Resolve the Trustodian user id for this API key
const userRef = await sdk.getSdkUserId();

// Wallet password comes from your authenticated user — never hardcode or persist it
const walletPassword = getWalletPasswordFromUserRequest();

// First-time setup: proves knowledge of the password to the API (OPRF + enc-token)
await sdk.setWalletPassword({ userRef, password: walletPassword });

// Create an EVM wallet — S0 is saved automatically to shareStore
const { wallets } = await sdk.createWallet({
  chain: EChainType.EVM,
  userRef,
  password: walletPassword,
  mode: 'generate',
});

console.log('New wallet address:', wallets[0].address);
```

<Warning>
  The wallet password shown here is for illustration only. **Never** hardcode it in source code,
  environment variables, or configuration files. **Never** log it or the device share. Pass it
  per-request from your authenticated user and call `lockWalletSession(userRef)` when the session
  ends. See [Authentication](/sdk-guide/authentication) for the full session lifecycle.
</Warning>

## Configuration Options

Pass a configuration object when you instantiate `TrustodianSDK`. The first three options are required for all key-touching operations.

| Option | Required | Default | Description |
| --- | --- | --- | --- |
| `apiKey` | **Yes** | — | Sent as `x-api-key` on every request |
| `network` | **Yes** | — | `'mainnet'` or `'testnet'` |
| `shareStore` | **Yes** | — | Your `IntegratorShareStore` for device share (S0) |
| `apiBaseUrl` | No | See below | Override the API host URL |
| `apiPathPrefix` | No | `/sdk` | HTTP path prefix |
| `timeout` | No | `30000` | Per-request timeout in milliseconds |
| `keyTtlMs` | No | `300000` | In-memory wallet-password session TTL (5 min) |
| `derivationUserId` | No | From `GET /sdk/users/me` | HKDF user-id override |
| `oprfPublicKeyHex` | No | Fetched at runtime | Pin the OPRF public key |
| `rpcOverrides` | No | `{}` | Per-chain custom RPC URLs |

**Default `apiBaseUrl` when omitted:**

When `apiBaseUrl` is not provided, the SDK connects to the Trustodian-managed API endpoint for the selected `network`. Contact your Trustodian account representative for the production endpoint URL, or override `apiBaseUrl` with a value provided by the Trustodian team for your environment.

You can override any of the advanced defaults as shown below:

```typescript
const sdk = new TrustodianSDK({
  apiKey: process.env.TRUSTODIAN_API_KEY!,
  network: 'testnet',
  shareStore: myShareStore,
  apiBaseUrl: 'https://your-api.example.com',
  keyTtlMs: 600_000,
  rpcOverrides: {
    evm: { 11155111: 'https://your-sepolia-rpc.com' },
  },
});
```
