## Overview

The **Address Book** lets you save and manage recipient addresses for quick reuse across transactions. Each entry stores an address, its blockchain type, a friendly name, and an optional memo (tag). When **Force Address Book** is enabled, only addresses saved in your address book can be used as transaction recipients.

## Address Types

Every address book entry is classified as one of two types:

| Type     | Description                                                  |
| -------- | ------------------------------------------------------------ |
| `Wallet` | A regular wallet address (self-custody, hardware wallet)     |
| `CEX`    | A centralized exchange address (Binance, Coinbase, etc.)     |

<Info>
  The address type helps identify whether a memo or destination tag is likely required.
  CEX addresses typically need a memo for proper fund routing.
</Info>

## Creating Entries

### Basic Entry

```typescript
const entry = await sdk.createAddressBookEntry({
  name: 'Alice Wallet',
  address: '0x5b927c461b71Ce25a74878dBe504492Ab80cbeE4',
  chainType: 'EVM',
  type: 'Wallet',
});

console.log('Saved:', entry.id, entry.name);
```

### Entry with Memo

For chains that support memos or destination tags, include the `memo` field:

```typescript
// XRP with destination tag
const xrpEntry = await sdk.createAddressBookEntry({
  name: 'Binance XRP',
  address: 'rN7n7otQDd6FczFgLdlqtyMVrRP3Jtxys',
  chainType: 'XRP',
  type: 'CEX',
  memo: '123456789',
});

// Solana with memo
const solEntry = await sdk.createAddressBookEntry({
  name: 'Exchange SOL',
  address: '5ZWj7a1f8tWkjBESHKgrLmXgEPXhRNMPE3m6tmFH4P6A',
  chainType: 'SOL',
  type: 'CEX',
  memo: 'user-deposit-ref',
});

// Tron with memo
const tronEntry = await sdk.createAddressBookEntry({
  name: 'Exchange TRX',
  address: 'TQCfX8tqKe5w8cUgBvUFV8nqVEJqHJbTBm',
  chainType: 'TRON',
  type: 'CEX',
  memo: '8827361',
});
```

<Warning>
  Each address can only be saved once per chain type. Attempting to create a duplicate entry returns a 409 Conflict error.
</Warning>

## Getting Entries

### List All Entries

```typescript
const entries = await sdk.getAddressBookEntries();

entries.forEach(entry => {
  console.log(`${entry.name}: ${entry.address} (${entry.chainType})`);
});
```

### Filter by Chain Type

```typescript
const evmEntries = await sdk.getAddressBookEntries({
  chainType: 'EVM',
});
```

### Filter by Address Type

```typescript
const cexEntries = await sdk.getAddressBookEntries({
  type: 'CEX',
});
```

### Combined Filters

```typescript
const solCexEntries = await sdk.getAddressBookEntries({
  chainType: 'SOL',
  type: 'CEX',
});
```

## Updating Entries

Update the name, memo, or both:

```typescript
// Update name
const updated = await sdk.updateAddressBookEntry({
  id: entry.id,
  name: 'Alice Main Wallet',
});

// Update memo
const updated = await sdk.updateAddressBookEntry({
  id: entry.id,
  memo: '999888777',
});

// Update both
const updated = await sdk.updateAddressBookEntry({
  id: entry.id,
  name: 'New Exchange Account',
  memo: '111222333',
});
```

### Clear a Memo

Set `memo` to `null` to remove an existing memo:

```typescript
const updated = await sdk.updateAddressBookEntry({
  id: entry.id,
  memo: null,
});

console.log(updated.memo); // null
```

<Info>
  At least one of `name` or `memo` must be provided when updating. Passing neither returns a 400 error.
</Info>

## Deleting Entries

```typescript
await sdk.deleteAddressBookEntry({ id: entry.id });
```

<Warning>
  When **Force Address Book** is enabled, deleting an entry means that address can no longer be used as a transaction recipient until it is re-added.
</Warning>

## Address Type Lookup

Check whether an address is known and what type it is:

```typescript
const result = await sdk.getAddressType({
  chainType: 'EVM',
  address: '0x5b927c461b71Ce25a74878dBe504492Ab80cbeE4',
});

console.log(result);
// {
//   type: 'Wallet',         // 'CEX', 'Wallet', or null
//   isInAddressBook: true,  // exists in your address book
//   isOnlyExist: false,     // Force Address Book enabled?
// }
```

| Field              | Type                       | Description                                      |
| ------------------ | -------------------------- | ------------------------------------------------ |
| `type`             | `'CEX'` \| `'Wallet'` \| `null` | Address type, or `null` if not in address book   |
| `isInAddressBook`  | `boolean`                  | Whether the address exists in your address book  |
| `isOnlyExist`      | `boolean`                  | Whether Force Address Book is enabled            |

## Force Address Book

When **Force Address Book** is enabled by the organization admin, all transaction recipients must exist in the sender's address book. This applies to both wallet `sendTransaction` and multisig `createOrder` / `createSendOrder` operations across all chains.

If a destination address is not found, the API returns:

```
400 Bad Request: "Destination address must be in your Address Book: <address>"
```

## Organization address book (SDK)

For **Organization SDK** routes under `sdk/organization`, `GET .../address-book` returns **all** corporate address book rows, including entries still pending email OTP verification. Each row includes `isVerified`. Only **`isVerified: true`** entries satisfy organization forced-address-book checks for outbound transfers; treat `isVerified: false` as metadata-only until the user completes OTP (or the row was added via API key with auto-verification).

Use **`PATCH sdk/organization/address-book/{addressId}`** (Master account, `AddressBookManagement` permission) to update the display label, memo, type, and Travel Rule fields. **`address` and `chainType` are immutable** after creation.

### Recommended Flow

Use `getAddressType` before sending to provide a good user experience:

```typescript
// 1. Check if address is in the address book
const check = await sdk.getAddressType({
  chainType: 'EVM',
  address: recipientAddress,
});

// 2. If Force Address Book is on and address not saved, prompt user to add it
if (check.isOnlyExist && !check.isInAddressBook) {
  const entry = await sdk.createAddressBookEntry({
    name: 'New Recipient',
    address: recipientAddress,
    chainType: 'EVM',
    type: 'Wallet',
  });
  console.log('Added to address book:', entry.id);
}

// 3. Proceed with the transaction
const result = await sdk.sendTransaction({
  chain: 'EVM',
  from: senderAddress,
  to: recipientAddress,
  amount: '0.5',
  chainId: 1,
  tokenAddress: '',
  decimals: 18,
  isNativeToken: true,
});
```

## Memo Support

Memos (also called destination tags or references) are used by certain blockchains and exchanges to identify the recipient of a transaction beyond just the address. This is especially important for CEX deposits where many users share a single deposit address.

### Chain Comparison

| Chain  | Memo Field Name     | Format        | Typical Use                   |
| ------ | ------------------- | ------------- | ----------------------------- |
| XRP    | Destination Tag     | Numeric       | CEX deposit routing           |
| SOL    | Memo                | String        | Exchange reference, indexing  |
| TRON   | Memo                | String        | CEX deposit routing           |

### Memo in Address Book

Store the memo alongside the address so it's automatically available for transactions:

```typescript
const entry = await sdk.createAddressBookEntry({
  name: 'Binance XRP',
  address: 'rN7n7otQDd6FczFgLdlqtyMVrRP3Jtxys',
  chainType: 'XRP',
  type: 'CEX',
  memo: '123456789',
});
```

### Memo in Transactions

When sending to an address that requires a memo, include it in the transaction:

```typescript
// Using createSendOrder (multisig)
const order = await sdk.createSendOrder({
  multisigId: multisig.id,
  walletAddress: myWalletAddress,
  to: recipientAddress,
  amount: '100',
  chainId: '1',
  memo: '123456789',
});

// Using sendTransaction (wallet)
const result = await sdk.sendTransaction({
  chain: 'XRP',
  from: senderAddress,
  to: recipientAddress,
  amount: '50',
  memo: '123456789',
  tokenAddress: '',
  decimals: 6,
  isNativeToken: true,
});
```

<AccordionGroup>
  <Accordion title="XRP">
    XRP uses a numeric **Destination Tag** to route funds to the correct account on exchanges.
    The tag is an unsigned 32-bit integer (0 to 4,294,967,295). Most exchanges require a
    destination tag for deposits. Sending without the correct tag may result in lost funds.
  </Accordion>

  <Accordion title="Solana">
    Solana memos are arbitrary strings attached to transactions via the SPL Memo program.
    Exchanges use them as deposit references. The memo is included as a separate instruction
    in the transaction.
  </Accordion>

  <Accordion title="Tron">
    Tron supports string-based memos attached to TRX and TRC-20 transfers. Exchanges use
    them for deposit identification. The memo is included in the transaction data field.
  </Accordion>
</AccordionGroup>

<Warning>
  Always verify memo requirements with the recipient. Sending funds to a CEX address without
  the required memo may result in lost or delayed deposits.
</Warning>
