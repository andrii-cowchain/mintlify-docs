## Server-Side Only

The SDK is designed for **integrator backends**. Your API key and end-user wallet passwords must never be exposed in browser or mobile client code.

## API Key

1. Sign in to the [Trustodian Dashboard](https://app.trustodian.com)
2. Go to **Settings** → **API Keys**
3. Create a key with the permissions your integration needs
4. Pass it to `TrustodianSDK` as `apiKey`

```typescript
import { TrustodianSDK, InMemoryShareStore } from 'vaults-multichain-sdk';

const sdk = new TrustodianSDK({
  apiKey: process.env.TRUSTODIAN_API_KEY!,
  network: 'testnet',
  shareStore: new InMemoryShareStore(),
});
```

## HTTP Headers (automatic)

The SDK sets these on every request:

| Header | When | Value |
| ------ | ---- | ----- |
| `x-api-key` | Always | Your API key |
| `Content-Type` | Always | `application/json` |

<Info>
  The web/mobile apps also use `x-device-id` for mobile sessions. **The Node SDK does not send
  `x-device-id`** — that header applies to native app JWT flows, not server-side SDK integration.
</Info>

## Wallet Password Session (required for crypto)

Before `createWallet`, `importWallet`, `sendTransaction`, `signOrder`, or any operation that reconstructs keys:

```typescript
const userRef = await sdk.getSdkUserId();

// First time
await sdk.setWalletPassword({
  userRef,
  password: userPassword,
  passwordHint: 'optional hint',
});

// Later sessions
await sdk.unlockWalletSession(userRef, userPassword);

// When done (or before idle timeout)
sdk.lockWalletSession(userRef);
```

| Method | Purpose |
| ------ | ------- |
| `setWalletPassword` | First-time setup + unlock |
| `unlockWalletSession` | Re-auth before crypto ops |
| `lockWalletSession` | Clear in-memory key material |
| `getEncTokenStatus` | Check if password is configured |
| `getWalletPasswordHint` | Read optional hint |

### Async context helper

If you omit `userRef` on a crypto call, wrap the operation in:

```typescript
await sdk.runWithUserRef(userRef, async () => {
  await sdk.sendTransaction({ chain: EChainType.EVM, from, to, amount: '0.1', chainId: 1, ... });
});
```

On Node.js the SDK uses `AsyncLocalStorage` so nested calls inherit the active `userRef`.

## `userRef`

`userRef` is the Trustodian user id for the API key owner. Resolve it once per session:

```typescript
const userRef = await sdk.getSdkUserId();
```

Use the same value for `setWalletPassword`, `unlockWalletSession`, `shareStore` keys, and crypto method calls. Call `getSdkUser()` when you need the full profile (`email`, `name`, etc.).

## Security Best Practices

<AccordionGroup>
  <Accordion title="Environment variables for API keys">
    ```typescript
    apiKey: process.env.TRUSTODIAN_API_KEY!
    ```
  </Accordion>
  <Accordion title="Never log or store wallet passwords">
    Collect the password from your user per session (server-side only). Do not hardcode it, put it in
    env vars, or save it in your database. The integrator persists only the **device share (S0)** via
    `shareStore` — treat that like key material too.
  </Accordion>
  <Accordion title="Implement IntegratorShareStore for production">
    `InMemoryShareStore` is for development only. Persist S0 per `(userRef, walletId)` in your DB.
  </Accordion>
  <Accordion title="Call lockWalletSession when idle">
    Default in-memory TTL is 5 minutes (`keyTtlMs`). Lock explicitly when a user session ends.
  </Accordion>
</AccordionGroup>

## Error Responses

| Status | Meaning |
| ------ | ------- |
| `401` | Invalid or missing API key |
| `403` | API key lacks permission |
| `400` | Invalid request body |
| `429` | Rate limited |

SDK errors extend `TrustodianBaseError` — use `isTrustodianError()` and typed classes like `UnauthorizedError`, `KeyMismatchError`.
