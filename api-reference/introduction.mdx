The Trustodian SDK REST API exposes every capability of the **vaults-multichain-sdk** as a set of authenticated HTTP endpoints. All routes are prefixed with `/sdk` and follow a consistent request/response pattern — making it straightforward to manage multisig wallets, assemble threshold-signature transactions, and query portfolio data across EVM, Solana, Bitcoin, Tron, Cardano, and XRP Ledger from any language or runtime. The API is the backbone that `TrustodianSDK` communicates with when you install the SDK package; you can also call these endpoints directly if you prefer raw HTTP.

## Base URLs

All SDK routes share a common `/sdk` path prefix. The table below shows the **default hosts used by `TrustodianSDK`** when you set `network` in the constructor. You can override either host by passing `apiBaseUrl` — useful for self-hosted deployments or local development proxies.

| `network` value | Default base URL |
|-----------------|------------------|
| `'mainnet'` | `https://stage-api.trustodian.online` |
| `'testnet'` | `https://afridax-backend-production.up.railway.app` |

A complete endpoint URL looks like:

```
https://stage-api.trustodian.online/sdk/wallets
```

<Note>
  The testnet base URL above is the SDK default for sandbox testing. For production use, always set `network: 'mainnet'` or supply your own host via `apiBaseUrl`.
</Note>

You can override the host at SDK initialisation time by passing `apiBaseUrl` in the `TrustodianSDK` constructor — useful for self-hosted deployments or local development proxies.

## Authentication

Every `/sdk/*` endpoint requires an API key issued from the [Trustodian Dashboard](https://app.trustodian.com). Pass the key in the `x-api-key` request header on every call.

| Header | Value |
|--------|-------|
| `x-api-key` | Your API key from the Trustodian Dashboard |

<Note>
  API keys are shown in full **only once** at creation time. Store yours securely — if lost, generate a new key from the Dashboard.
</Note>

You can scope keys with per-method permissions (e.g. `view_balances` only), restrict them to an IP whitelist, and toggle them on or off — all without revoking the key permanently. Manage your API keys from the [Trustodian Dashboard](https://app.trustodian.com).

## Non-Custodial Endpoints

<Info>
  The Trustodian SDK is **non-custodial** — private key material never leaves your infrastructure in plain text. The server holds only an encrypted share.
</Info>

Most `/sdk/*` routes are **read-only** — listing wallets, fetching portfolio balances, querying order history — and require only the `x-api-key` header.

A small subset of routes participates in the **non-custodial signing flow** and should be called through the `TrustodianSDK` package rather than raw HTTP:

| Route pattern | Purpose |
|---------------|---------|
| `GET /sdk/wallets/share/c/…` | Retrieve the encrypted **client** key share |
| `GET /sdk/wallets/share/s/…` | Retrieve the encrypted **server** key share |
| `POST /sdk/users/enc-token` | Exchange credentials for an encryption token |
| `POST /sdk/oprf/evaluate` | OPRF evaluation step in the key-hardening protocol |

The SDK handles share decryption, OPRF blinding/unblinding, and threshold-signature assembly transparently. See the [Key Management guide](/sdk-guide/key-management) for the full protocol walkthrough.

Organization-scoped routes live under `/sdk/organization/…` (team management, wallet assignments, vaults, orders). Start with [Invite Member](/api-reference/organizationsdk/post-organization-team-invite) or the [Organization guide](/sdk-guide/organization).

## Code Examples

<CodeGroup>

```javascript JavaScript
fetch('https://stage-api.trustodian.online/sdk/wallets', {
  method: 'GET',
  headers: {
    'x-api-key': 'YOUR_API_KEY',
    'Content-Type': 'application/json',
  },
})
  .then(response => response.json())
  .then(data => console.log(data));
```

```bash cURL
curl -X GET "https://stage-api.trustodian.online/sdk/wallets" \
  -H "x-api-key: YOUR_API_KEY" \
  -H "Content-Type: application/json"
```

```python Python
import requests

headers = {
    "x-api-key": "YOUR_API_KEY",
    "Content-Type": "application/json",
}

response = requests.get(
    "https://stage-api.trustodian.online/sdk/wallets",
    headers=headers,
)
data = response.json()
print(data)
```

</CodeGroup>

## Supported Chains

The Trustodian SDK API supports threshold multisig across six blockchain ecosystems.

<CardGroup cols={2}>
  <Card title="EVM Chains" icon="ethereum">
    Ethereum, BNB Smart Chain, Polygon, Avalanche, and any EVM-compatible
    network. Uses **GnosisSafe** multisig under the hood.
  </Card>
  <Card title="Solana" icon="sun">
    Full Solana mainnet and devnet support via the **Squads** multisig program.
    Supports SOL and SPL token transfers.
  </Card>
  <Card title="Bitcoin" icon="bitcoin">
    Native Bitcoin P2WSH multisig. Orders return a **PSBT** for incremental
    owner signing before broadcast.
  </Card>
  <Card title="Tron" icon="bolt">
    TRX and TRC-20 token transfers through Tron's native multisig permission
    layer.
  </Card>
  <Card title="Cardano" icon="eye">
    ADA and native asset transfers using Cardano **native script** multisig.
    Follows the EUTXO model — signatures are `VkeyWitness` objects collected
    off-chain.
  </Card>
  <Card title="XRP Ledger" icon="x">
    XRP native multisig via **SignerList**. Supports sequential transaction
    execution with account sequence management.
  </Card>
</CardGroup>

## Core Workflow

Every chain follows the same five-step lifecycle. The specific endpoint names differ per chain, but the pattern is identical.

<Steps>
  <Step title="Authenticate">
    Obtain an API key from the [Trustodian Dashboard](https://app.trustodian.com)
    and include it as `x-api-key` on every request.
  </Step>
  <Step title="Create or Fetch a Wallet">
    Call `GET /sdk/wallets` to list existing multisig wallets, or use the
    wallet-creation endpoints to deploy a new one on your target chain.
  </Step>
  <Step title="Create an Order">
    Submit a transaction intent via the chain-specific `POST /sdk/order/create/{chain}`
    endpoint. The API returns the unsigned transaction data (PSBT for Bitcoin,
    tx body hash for Cardano, Squads params for Solana, etc.).
  </Step>
  <Step title="Collect Threshold Signatures">
    Each authorised wallet owner calls `POST /sdk/order/sign/{chain}` with
    their partial signature. When the configured threshold is reached, the
    order status advances to **ready**.
  </Step>
  <Step title="Execute the Transaction">
    Call `POST /sdk/order/execute/{chain}/{id}` to broadcast the fully signed
    transaction to the blockchain. The order status transitions from **ready** →
    **execution** → **executed** once confirmed on-chain.
  </Step>
</Steps>
